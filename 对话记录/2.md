## A. 项目总览

### 1. 项目名称

OpenFOAM Python 桌面仿真客户端，暂定内部代号 `FoamDesk`。

### 2. 项目愿景

做一个运行在 WSL/Linux 图形环境中的 Python 桌面客户端，让用户不必直接手写大量 OpenFOAM case 文件和命令，也能完成常见 CFD 仿真任务。它不是替代 OpenFOAM，而是作为 OpenFOAM 的图形化工程工作台。

### 3. 产品目标

- 让用户可以通过桌面客户端创建、管理、运行 OpenFOAM 仿真项目
- 降低 OpenFOAM 入门门槛，减少命令行和字典文件编辑成本
- 在一个统一界面中完成项目管理、参数配置、任务执行、日志查看和结果浏览
- 为后续扩展更多求解器、模板、物理模型保留清晰结构

### 4. 用户画像

- 初级 CFD 学习者：知道仿真目标，但不熟悉 OpenFOAM 目录和命令
- 工程研发人员：会使用部分 OpenFOAM 求解器，希望减少重复操作
- 内部工具用户：需要一个可维护、可交付、可复用的本地仿真客户端

### 5. 核心使用场景

1. 新建一个仿真项目，并选择一个 case 模板
2. 修改基础参数，如流体属性、边界条件、求解控制参数
3. 执行 `blockMesh`、`snappyHexMesh`、`simpleFoam` 等命令
4. 在软件中实时查看日志、状态、错误信息
5. 查看结果文件、提取残差等关键指标
6. 管理多个项目和多次运行历史

### 6. MVP 范围

- 项目工作区管理
- 单机本地 OpenFOAM 环境探测
- 基于模板创建 case
- 常用字典文件表单化编辑
- 后台执行网格与求解命令
- 实时日志查看和任务状态管理
- 基础结果浏览：残差曲线、输出文件列表、关键文本结果

### 7. 非 MVP 范围

- 完整三维高级交互式后处理替代 ParaView
- 多机分布式任务调度
- 云端协同、账号体系、远程任务管理
- 完整 CAD 建模能力
- 覆盖所有 OpenFOAM 求解器和高级物理模型

### 8. 成功标准

- 新手能在 GUI 中完成一个标准模板 case 的创建、参数编辑、运行和结果查看
- 关键流程无需用户手动切换到命令行
- 失败时能给出可诊断信息，而不是静默失败
- 工程结构足够清晰，能继续扩展求解器和模板

## C. 推荐技术栈与选型理由

### 一人可落地推荐栈

- GUI：PySide6
- 图表：matplotlib
- 结果快速交互图：pyqtgraph，作为增强项
- 三维后处理集成：先不内嵌，MVP 阶段调用外部 ParaView 或读取结果摘要
- 本地数据库：SQLite
- 配置文件：YAML 用于静态模板与应用设置，JSON 用于轻量运行元数据
- 日志：Python `logging`
- 任务执行：Qt 主线程 + `QThread` 工作线程，内部使用 `subprocess.Popen`
- 测试：pytest + pytest-qt
- 代码质量：ruff + black + mypy
- 打包：开发阶段先源码运行，MVP 后评估 PyInstaller；若性能和可执行质量要求高再看 Nuitka

### 为什么选 PySide6

- Qt 的桌面应用能力足够强，适合复杂工作区式界面
- 对 Linux/WSL GUI 适配比 Tkinter 更稳，生态更成熟
- 信号槽、线程、模型视图结构适合工程软件
- PySide6 LGPL 许可对个人和后续商业化更稳妥

### 不优先选其他方案的原因

- Tkinter：上手快，但复杂工程 UI、模型视图、扩展能力不够理想
- wxPython：可用，但中文资料、现代生态、团队常见经验不如 Qt
- PyQt：技术上可以，但许可约束相对更敏感
- asyncio：不适合直接承担这类 GUI 中的重进程调用主模型
- multiprocessing：进程隔离强，但会显著增加状态同步复杂度，MVP 阶段没必要
- TinyDB：轻量，但任务记录、查询、后续扩展不如 SQLite 稳

## D. 客户端系统架构设计

### 分层结构

```text
UI 展示层
  -> 应用控制层
    -> 业务逻辑层
      -> OpenFOAM 集成层
      -> 文件管理层
      -> 数据存储层
      -> 日志与监控层
        -> Linux 本地进程调用层
```

### 各层职责

- UI 展示层
  负责窗口、面板、表单、按钮、列表、树、图表显示，不直接拼命令，不直接操作 OpenFOAM。
- 应用控制层
  接收 UI 动作，组织用例执行，协调服务调用，处理界面状态切换。
- 业务逻辑层
  维护项目、case、任务、结果这些核心领域对象和规则。
- OpenFOAM 集成层
  负责命令构造、环境加载、字典文件读写、结果提取。
- Linux 本地进程调用层
  统一封装 `subprocess`，处理 `bash -lc`、工作目录、环境变量、stdout/stderr、终止和超时。
- 文件管理层
  统一管理工作区目录、模板复制、结果文件归档、路径校验。
- 数据存储层
  保存项目信息、任务历史、设置项、结果索引。
- 日志与监控层
  保存应用日志、任务日志、诊断信息，向 UI 提供状态和日志流。

### 必须解耦的点

- UI 不能直接调用 `subprocess`
- UI 不能直接读写 OpenFOAM 字典文件
- OpenFOAM 命令组装不能散落在按钮回调里
- 存储层不能依赖具体窗口类
- 结果解析逻辑不能直接耦合图表控件

## E. 模块级详细设计

- 模块职责
  管理模板清单、从模板复制 case、校验模板完整性。
- 主要类设计

  模板发现、模板复制、缺文件报错。

### 3. 参数配置模块

- 模块职责
  将表单字段映射为 OpenFOAM 字典配置。
- 主要类设计
  `ParameterSchema`、`ParameterService`、`FormMapper`
- 测试点
  默认值加载、参数校验、保存回写。

### 4. OpenFOAM 配置文件生成模块

- 模块职责
  生成和更新 `controlDict`、`fvSchemes`、`fvSolution` 等。
- 主要类设计
  `FoamDictWriter`、`ControlDictBuilder`、`TransportPropertiesBuilder`

  文件生成格式、注释保留策略、覆盖更新。

### 5. Linux 本地命令调用模块

- 模块职责
  统一执行本地命令并输出实时日志。
- 主要类设计
  `CommandRunner`、`ProcessHandle`、`ShellCommandBuilder`
- 测试点
  stdout/stderr 捕获、退出码判断、取消任务。

### 6. 仿真任务执行模块

- 模块职责
  定义网格生成、求解、后处理等任务生命周期。
- 主要类设计
  `SimulationTaskService`、`TaskOrchestrator`、`TaskStateMachine`
- 状态流转
  `PENDING -> RUNNING -> SUCCEEDED / FAILED / CANCELED`
- 测试点
  单任务执行、串行任务链、失败回滚策略。

### 7. 任务状态与日志跟踪模块

- 模块职责
  汇总任务状态，推送日志到 UI，并保存历史记录。
- 主要类设计
  `TaskLogService`、`TaskEventBus`、`TaskHistoryRepository`

### 8. 结果文件管理模块

- 模块职责
  管理时间步目录、后处理输出、文件索引和结果摘要。
- 主要类设计
  `ResultService`、`ResultIndexBuilder`

### 9. 后处理与可视化模块

- 模块职责
  绘制残差曲线、展示 probes/forces 文本结果、提供外部 ParaView 打开入口。
- 主要类设计
  `ResidualParser`、`ResidualPlotPresenter`、`PostProcessService`

### 10. 本地配置与设置模块

- 模块职责
  管理工作区路径、OpenFOAM 环境脚本路径、默认求解器、UI 偏好。
- 主要类设计
  `AppSettingsService`、`SettingsRepository`

### 11. 错误处理与诊断模块

- 模块职责
  统一错误分类、提示、诊断建议和导出日志。
- 主要类设计
  `ErrorClassifier`、`DiagnosticService`

## I. OpenFOAM 集成设计

### 核心原则

- 只通过统一的 OpenFOAM 集成层调用命令
- 每次任务明确指定 case 目录和环境脚本
- 所有日志都实时流式采集
- 每个任务实例都带唯一运行 ID，避免相互污染

### 命令执行模型

推荐封装为：

```text
source <OpenFOAM_env.sh> && cd <case_dir> && <foam_command>
```

GUI 中不直接依赖用户当前 shell，会由 `CommandRunner` 统一使用：

```text
bash -lc "source ... && cd ... && blockMesh"
```

### MVP 先做的命令

- `blockMesh`
- `simpleFoam`
- `checkMesh`
- `paraFoam` 的外部启动入口

### 第二阶段再做

- `snappyHexMesh`
- `pisoFoam`
- `foamToVTK`
- forces/probes 自动化提取增强

### 配置抽象策略

- MVP 阶段只抽象高频字段，不追求覆盖全部字典语法
- 表单层维护参数 schema
- 写文件层根据 schema 渲染标准 OpenFOAM 字典
- 高级模式允许“查看生成文件”和“手动补充自定义字段”

### 最易踩坑的点

- 不同发行版或安装方式的 OpenFOAM 环境脚本路径不同
- GUI 进程环境不等于交互式 shell 环境
- 日志中既有 stdout 又有 stderr，不能只看一边
- 求解器退出码和日志报错要结合判断
- 多任务并发时不能共用同一个 case 输出目录

## N. 现在就开始：PRD v1 + 技术方案 v1 + 模块拆分 v1 + 第一阶段实施建议

### 1. PRD v1

#### 产品定义

这是一个运行于 WSL/Linux 图形环境中的 Python 桌面客户端，用于将 OpenFOAM 的 case 管理、参数配置、命令执行和结果查看图形化。

#### 核心问题

- OpenFOAM 初学者难以上手
- 工程师重复编辑大量字典文件
- 命令行执行过程缺少统一可视化管理
- 项目、运行记录和结果管理混乱

#### 核心价值

- 降低学习与使用门槛
- 提升标准流程执行效率
- 减少人为路径和命令错误
- 为后续模板化与标准化打基础

#### 功能需求清单

- 项目创建与打开
- 工作区管理
- 模板创建 case
- 参数编辑
- OpenFOAM 命令执行
- 日志查看
- 结果文件管理
- 残差结果展示
- 基础设置与环境检查

#### 非功能需求清单

- 性能
  普通中小 case 执行期间 UI 不冻结
- 稳定性
  命令失败时必须给出可见错误状态
- 可维护性
  严格分层，避免业务逻辑散落 UI
- 可扩展性
  可新增模板、求解器、参数 schema
- 易用性
  新手可以通过默认模板跑通
- 安全性
  禁止任意命令拼接执行，严格限制工作目录和命令构造

### 2. 技术方案 v1

#### 技术决策

- 使用 PySide6 作为 GUI 框架
- 使用 SQLite 管理项目索引和任务历史
- 使用 YAML 管理应用设置和参数 schema
- 使用 `subprocess.Popen` + `QThread` 实现后台任务
- 使用 `bash -lc` 加载 OpenFOAM 环境并执行命令
- 使用 matplotlib 绘制 MVP 阶段的残差曲线

#### 关键架构约束

- 所有 UI 动作只能进入控制层，不允许直调 OpenFOAM 命令
- OpenFOAM 集成层只关心 case、命令、环境、结果，不关心界面
- 文件系统与数据库都必须通过仓储或服务封装访问

### 3. 模块拆分 v1

- `app`
  应用启动、依赖组装、全局配置
- `ui`
  窗口、页面、组件
- `controllers`
  响应 UI 事件，编排服务
- `domain`
  项目、任务、结果等实体
- `services`
  项目管理、任务管理、设置管理
- `integrations.openfoam`
  OpenFOAM 命令、字典生成、结果解析
- `infrastructure.process`
  本地命令执行
- `infrastructure.storage`
  SQLite、YAML、JSON、文件系统访问
- `infrastructure.logging`
  应用日志和任务日志
