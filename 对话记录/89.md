## 阶段 71：求解器准备页分层与一键启动链路修复

当前阶段目标：
- 按用户要求把完整仿真缺失项中的 1-4 放到 `求解器准备` 页面。
- 1-4 分别是：`snappyHexMeshDict`、`controlDict`、`fvSchemes`、`fvSolution`。
- 后续再把 5-6，也就是 `0/U`、`0/p` 和 `physicalProperties` 分离到 `仿真准备` 页面。
- 修复后面的 `一键启动仿真` 按钮，让它能自动衔接前面绘制几何生成的 STL 和 dict。

本阶段完成功能：
- 原 `仿真准备` tab 改为 `求解器准备`。
- 求解器准备页的说明改为只聚焦 1-4。
- 求解器准备页的按钮 `补齐仿真文件` 改为 `补齐求解器文件`。
- 求解器准备页状态检查现在优先显示 `snappyHexMeshDict/controlDict/fvSchemes/fvSolution`。
- 如果当前 Case 有 STL，`snappyHexMeshDict` 会作为求解器准备必需项显示。
- 如果当前 Case 没有 STL，流水线允许跳过 `snappyHexMesh`，不再强制要求 STL。
- 一键仿真会自动使用绘制几何导出的 `body_*.stl`，不再要求用户手动选择 STL。
- 如果有绘制几何生成的多个 STL，优先自动选择最新的 `body_*.stl`。
- `求解运行` 页和 `求解器准备` 页的一键启动按钮已经统一到同一条 `_run_simulation_pipeline()`。

本阶段页面与 UI 完成情况：

求解器准备页 `SolverPrepareTab`：
- 页面标题：`求解器准备`，使用 `QLabel` 渲染。

本阶段完成设计：
- 新增自动 STL 选择逻辑 `_auto_stl_asset_for_snappy()`。
- `_select_stl_asset_for_snappy()` 优先复用自动选择逻辑，只有无法判断时才保留人工选择能力。
- `_run_simulation_pipeline()` 不再一开始强制用户选择 STL。
- `_run_simulation_pipeline()` 改为：有 STL 时自动生成 `snappyHexMeshDict` 并运行 `snappyHexMesh`；无 STL 时跳过 `snappyHexMesh`。
- `_run_full_pipeline()` 保持兼容，但内部委托 `_run_simulation_pipeline()`，避免旧按钮再次走错误逻辑。

本阶段数据流设计：
- 绘制几何页 -> `生成 blockMeshDict + STL` -> 输出 `system/blockMeshDict` 和 `constant/triSurface/body_*.stl`。
- 求解器准备页 -> `补齐求解器文件` -> 自动识别 `body_*.stl` -> 生成 `snappyHexMeshDict` -> 保存 `controlDict/fvSchemes/fvSolution`。
- 求解器选择页/仿真参数页 -> 保存 solver 和参数。
- 求解运行页 -> `一键启动仿真` -> 自动识别 STL -> 有 STL：`blockMesh -> snappyHexMesh -> checkMesh -> solver`。
- 求解运行页 -> `一键启动仿真` -> 没有 STL：`blockMesh -> checkMesh -> solver`。
