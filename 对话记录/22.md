# Sprint 3：OpenFOAM 最小仿真执行闭环

当前阶段目标：
- 把项目从“只能生成 case 和执行 blockMesh”推进到“能执行一个最小 OpenFOAM 仿真”。
- 本阶段只做内置 cavity 风格最小算例，先跑通 `blockMesh + icoFoam`，不做复杂几何导入、snappyHexMesh、ParaView 后处理和参数表单。

本阶段完成功能：
- 新建项目时自动生成可运行的 OpenFOAM 最小 case 文件。
- 生成 `system/blockMeshDict`，用于创建一个 1x1x1 的基础网格。
- 生成 `0/U`、`0/p`、`constant/transportProperties`、`constant/physicalProperties`、`system/controlDict`、`system/fvSchemes`、`system/fvSolution`。
- 兼容 OpenFOAM-dev 的 `icoFoam` 属性文件要求，新增 `physicalProperties`。
- 在 `fvSolution` 的 `PISO` 中配置 `pRefCell/pRefValue`，避免压力场缺少参考值导致求解失败。
- 顶部“运行”按钮从单独执行 `blockMesh` 调整为执行 `blockMesh && icoFoam`。
- “求解器 -> 运行最小仿真”菜单项调用同一套最小仿真流程。
- 运行页和日志区显示当前执行流程：`blockMesh -> icoFoam`。

本阶段页面与 UI 完成情况：

主窗口 `MainWindow`：
- 页面标题：保持 `FoamDesk` 主窗口标题。
- UI 组件：继续使用 VS Code 风格菜单栏、工具栏、资源管理器、标签页、底部日志/任务面板和状态栏。
- 按钮或菜单项：工具栏“运行”可点击，触发 `_run_minimal_simulation()`；菜单“求解器 -> 运行最小仿真”可点击，触发同一逻辑。
- 键盘操作：本阶段未新增快捷键。
- 鼠标操作：点击“运行”后切换到“求解运行”页和底部“任务”面板。
- 状态提示：状态栏显示“最小仿真运行中/完成/失败/已停止”，日志面板输出 OpenFOAM 命令实时日志。

本阶段完成设计：
- `ProjectService` 负责生成最小 OpenFOAM case 模板，UI 不直接拼写 OpenFOAM 字典文件。
- `MainWindow` 只负责组织用户操作、状态展示和调用命令执行器。
- OpenFOAM 命令执行仍通过既有 `OpenFoamService` 和 `CommandRunner` 处理，避免 UI 线程阻塞。
- 本阶段用 cavity 风格算例验证 OpenFOAM 链路，后续再扩展为“风吹汽车/物体绕流”等更复杂核心场景。

本阶段数据流设计：
- 用户点击“新建项目” -> `ProjectService.create_project()` -> 生成项目目录和最小 OpenFOAM case 文件 -> 刷新项目树。
- 用户点击“运行” -> `MainWindow._run_minimal_simulation()` -> 校验当前项目和 `system/blockMeshDict` -> 调用 `OpenFoamService.run_command()`。
- `OpenFoamService` -> `CommandRunner` -> source OpenFOAM 环境脚本 -> 在当前 case 目录执行 `blockMesh && icoFoam` -> 实时回传 stdout/stderr。
- 命令输出 -> `MainWindow._handle_command_output()` -> 写入日志面板和状态栏。
- 命令完成 -> `_handle_command_finished()` -> 更新任务状态为完成或失败。

本阶段核心业务场景完成进度：
- 项目管理业务场景：已能创建带最小 case 模板的项目。
- OpenFOAM 执行业务场景：从 `blockMesh` 单命令推进到 `blockMesh + icoFoam` 最小仿真链路。
- 参数配置业务场景：仍是占位，尚未把速度、粘度、时间步长做成可编辑表单。
- 日志业务场景：已能展示命令实时输出。
- 后处理业务场景：未完成，后续需要解析残差、结果文件和可视化。
- UI 组件化业务场景：未新增组件，沿用现有主窗口结构。
- 未完成内容：复杂几何导入、snappyHexMesh、参数表单、结果可视化、报告导出留到后续 Sprint。
