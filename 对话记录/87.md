## 阶段 69：求解器运行接入、湍流文件与 patch 边界生成

当前阶段目标：
- 让 `求解器选择` 页面不再只是 UI 配置，而是实际影响运行命令。
- 补齐 `simpleFoam/pisoFoam` 和湍流模型所需的基础字段文件。
- 将边界条件从单纯全局参数推进到按 patch 名称自动生成边界字段。
- 本阶段做可运行 MVP：自动按 patch 名称识别入口、出口、壁面、对称面；后续再做可视化逐 patch 编辑表。

本阶段完成功能：
- 一键仿真流水线会读取当前 Case 的 `solver_name`，实际运行 `icoFoam`、`simpleFoam` 或 `pisoFoam`。
- 最小仿真入口也会按当前选择的求解器运行，不再固定写死 `icoFoam`。
- 保存仿真参数时自动生成求解器支持文件：
  - `0/U`
  - `0/p`
  - `constant/turbulenceProperties`
  - `0/k`
  - `0/epsilon`
  - `0/nut`
  - `system/foamdesk_patch_boundary_config.json`
- 当湍流模型为 `laminar` 时生成 laminar `turbulenceProperties`。
- 当湍流模型为 `RAS kEpsilon` 时生成 `k/epsilon/nut` 字段和 `RAS kEpsilon` 配置。
- 按 patch 名称自动识别：
  - `inlet` 或 `movingWall`：入口类。
  - `outlet`：出口类。
  - `symmetry`：对称面。
  - 其他 patch：壁面/物体表面。
- 一键 snappy 仿真流水线中，snappy 后的 `movingWall/fixedWalls/importedGeometry` 也会生成完整求解器支持字段。

本阶段完成设计：
- `ProjectService.ensure_solver_support_files()` 负责生成当前 solver 所需基础文件。
- `ProjectService` 新增 patch role 识别逻辑，将 patch 自动归类为 inlet/outlet/wall/symmetry。
- `U/p/k/epsilon/nut` 的边界条件由统一的 patch role 规则生成，避免只同步 `U/p` 导致湍流字段缺失。
- `MainWindow._run_simulation_pipeline()` 和 `_run_minimal_simulation()` 读取 `case_parameter_service.load()` 的 `solver_name` 后拼接实际命令。
- 保留另一个 agent 已留下的页面精简方向：旧 `几何/CAD` 标签已从工作区移除，相关 tab 索引跟随调整。

本阶段数据流设计：
- 用户选择 solver -> 保存参数 -> `case_parameter_service.save()` 写入基础字典 -> `project_service.ensure_solver_support_files()` 生成 U/p/turbulence/k/epsilon/nut/patch JSON。
- 用户点击最小仿真 -> `_save_case_parameters()` -> 读取 `solver_name` -> 执行 `blockMesh && {solver_name}`。
- 用户点击一键仿真流水线 -> 生成 snappyHexMeshDict -> 生成 snappy 后 patch 对应字段 -> 执行 `blockMesh -> snappyHexMesh -> checkMesh -> {solver_name}`。
- snappy 后 patch 名称 -> `ensure_solver_support_files(..., ("movingWall", "fixedWalls", "importedGeometry"))` -> 生成完整 U/p/k/epsilon/nut 边界字段。
