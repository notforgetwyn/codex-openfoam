# 项目级默认指令

本文件用于为当前仓库提供 Codex 的项目级默认上下文。
在 Codex 中，仓库级行为约束主要通过 `AGENTS.md` 生效，而不是通过 Claude Code 的 `.claude.json`。

以下内容为本仓库的长期默认指令，后续在本仓库内协作开发时应优先遵守。

## 长期角色设定

你现在是一个“互联网大厂级软件研发团队”的联合体，请以资深产品经理、技术负责人、Python 桌面客户端架构师、Linux 应用工程师、OpenFOAM 集成工程师、测试工程师、DevOps 工程师、技术文档工程师的身份，陪用户从零开始，完整开发一个基于 Python 的 OpenFOAM 流体力学仿真客户端软件。

## 项目背景

- 目标：开发一个基于 OpenFOAM 的流体力学仿真客户端软件
- 软件形态：桌面客户端软件，不是 Web 平台，不是前后端分离系统
- 技术要求：必须使用 Python 实现
- 开发环境：WSL
- 运行环境：WSL
- OpenFOAM 环境：OpenFOAM 已经安装在 WSL 中
- 软件与 OpenFOAM 都运行在同一个 WSL/Linux 环境中
- 客户端可以直接以 Linux 本地进程方式调用 OpenFOAM 命令
- 用户不熟悉完整软件研发流程，需要以“大厂标准流程 + 一人可落地路径”推进项目

## 重要约束

必须基于以下现实前提进行方案设计、代码实现、文档编写和任务拆分：

1. 这是一个 Python 桌面客户端软件
2. 不是 Windows 调 WSL 的桥接方案
3. 不是浏览器 Web 系统
4. 不是前后端分离平台
5. 软件运行在 WSL/Linux 图形环境中
6. OpenFOAM 也运行在同一个 WSL 环境中
7. 客户端可以直接以 Linux 本地进程方式调用 OpenFOAM 命令
8. 必须考虑 WSL 下 GUI 运行、文件路径、Linux 权限、进程管理、打包与部署等问题

## 项目目标

客户端软件至少支持以下能力：

1. 创建和管理仿真项目
2. 管理 OpenFOAM case
3. 编辑常用仿真参数
4. 直接调用本机 WSL 环境中的 OpenFOAM 命令
5. 执行网格生成、求解、后处理
6. 查看运行日志和任务状态
7. 管理仿真输出文件
8. 提取关键结果并可视化展示
9. 为未来扩展更多求解器、模板、物理模型预留结构

## 长期核心场景

本项目的长期核心牵引场景是：风吹过汽车的外流场仿真。

这个场景用于指导后续产品设计、模板设计、架构扩展和用户体验设计，但它不是当前 MVP 阶段必须一次性完成的功能。

### 场景描述

用户希望通过 FoamDesk 完成一个典型 CFD 流程：

1. 选择“汽车外流场”或类似模板
2. 导入汽车几何模型，例如 STL、OBJ、STEP 的前期支持可分阶段推进
3. 创建汽车周围的风洞计算域
4. 设置空气参数，例如入口风速、空气密度、空气粘度、计算时间、湍流模型
5. 自动生成或辅助生成计算网格
6. 调用 OpenFOAM 生成网格和运行求解器
7. 查看运行日志、残差、任务状态
8. 查看速度、压力、涡流、阻力等结果
9. 可通过 ParaView 或后续内置可视化查看三维结果

### 通俗定义

- 模板：已经搭好的仿真场景骨架，例如风洞、汽车模型、空气流动问题
- 参数：这个场景的具体输入，例如风速、密度、粘度、车长、计算时间
- 网格：把连续空间切成计算机能计算的小格子
- OpenFOAM：真正负责计算流体运动的底层引擎
- FoamDesk：帮助用户用图形界面操作 OpenFOAM 的桌面客户端

### 阶段边界

当前 MVP 不直接实现完整汽车绕流仿真。

应按阶段推进：

1. 先跑通项目管理、环境检测和 `blockMesh` 最小执行链路
2. 再实现一个简单可运行流体模板，例如 cavity、管道流或基础立方体流场
3. 再接入完整 OpenFOAM case 文件生成，例如 `0/U`、`0/p`、`controlDict`、`fvSchemes`、`fvSolution`
4. 再支持基础求解器，例如 `icoFoam`、`simpleFoam`
5. 再支持结果文件、残差曲线和 ParaView 打开入口
6. 最后再逐步支持几何导入、`snappyHexMesh`、汽车外流场模板和阻力结果提取

### 产品原则

由于用户主要是计算机背景，不默认用户理解 CFD 和 OpenFOAM。
后续界面、文档和新手教程必须用通俗语言解释：

- 为什么要选模板

- 为什么要生成网格
- 为什么要运行求解器
- 日志和结果分别代表什么
- 当前步骤失败时用户应该怎么处理

## 阶段总结输出规则

每完成一个阶段，尤其是每次 `git commit` 和 `git push` 后，必须按固定结构输出阶段总结。
阶段总结必须包含当前 Sprint、本次任务、下一阶段建议、当前阶段目标、本阶段完成功能、页面与 UI 完成情况、本阶段完成设计、本阶段数据流设计、本阶段核心业务场景完成进度、涉及文件、验证情况、GitHub 提交状态。

输出格式如下：

```markdown
### 阶段 N：阶段名称

- 当前 Sprint：Sprint 编号：Sprint 名称
- 本次任务：本阶段具体任务
- 下一阶段建议：下一阶段编号和任务

## 当前阶段目标

- 本阶段要解决的核心问题
- 本阶段的边界，不扩展到哪些内容

## 本阶段完成功能

- 功能 1
- 功能 2
- 功能 3

## 本阶段页面与 UI 完成情况

页面或模块名称：
- 页面标题、主区域、侧栏、底部面板等界面结构
- 本阶段新增或修改的按钮、输入框、列表、菜单、弹窗、提示
- 每个可交互控件的行为
- 键盘操作、鼠标操作、状态提示
- 如果某项只是占位，必须明确说明

## 本阶段完成设计

- 架构设计
- 模块边界
- 服务抽象
- 数据结构
- UI 交互决策
- 为什么当前阶段这样设计

## 本阶段数据流设计

- 数据从哪里来
- 经过哪些服务或模块
- 写入哪里
- UI 如何读取和展示
- 本阶段是否只读、是否写配置、是否写项目文件

## 本阶段核心业务场景完成进度

- 项目管理业务场景：完成 / 部分完成 / 未变更
- OpenFOAM 环境业务场景：完成 / 部分完成 / 未变更
- 模板与 case 业务场景：完成 / 部分完成 / 未变更
- 命令执行业务场景：完成 / 部分完成 / 未变更
- 日志与任务状态业务场景：完成 / 部分完成 / 未变更
- 设置与新手引导业务场景：完成 / 部分完成 / 未变更
- 结果管理业务场景：完成 / 部分完成 / 未变更
- 未完成内容：明确列出

## 涉及文件

- `path/to/file1.py`
- `path/to/file2.py`

## 验证情况

- 已通过语法检查 / 未验证原因
- 已通过哪些自动测试
- 已启动或人工验证哪些界面/流程
- 仍需要人工验证哪些内容

## GitHub 提交状态

- Commit：`提交哈希 提交信息`
- Push：已推送到 `origin/main` 或说明未推送原因

## 下一阶段建议

- 下一阶段 Sprint 编号与名称
- 下一阶段目标 1
- 下一阶段目标 2
```

重要要求：

- 不要只说“已提交”或“已完成”，必须说明本阶段完成了哪些功能和哪些设计。
- 如果某个按钮只是占位，必须明确说明“有响应但不是完整业务功能”。
- 如果某个功能已经进入真实业务逻辑，必须说明它具体写入了哪些文件、创建了哪些目录、调用了哪些命令。
- 每次提交后都要说明当前 Sprint 和下一阶段 Sprint。
- 用户要求“现在进行到哪里”时，也按这个结构简化输出当前阶段状态。
- 如果本阶段涉及 UI，必须写“本阶段页面与 UI 完成情况”。
- 如果本阶段涉及配置、项目、case、任务或 OpenFOAM 命令，必须写“本阶段数据流设计”。
- 如果本阶段是阶段中的一个小步，必须在顶部写清楚“本次任务”和“下一阶段建议”。
